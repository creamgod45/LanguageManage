# 在地化管理器需求規格

## 1. 目標

建立一個 JetBrains IDE 插件，用來列管 PHP Laravel i18n、一般 JSON、YAML 及 JetBrains/Java ResourceBundle Properties 語言檔。使用者必須自行建立方案並明確選取檔案；不同方案彼此隔離，插件不得自動納管專案檔案。

## 2. 支援環境

- 插件最低相容 JetBrains Platform build `253.5`（IntelliJ IDEA 2025.3.5），以 2025.3.5 作為編譯與測試基準，且不設定 `until-build`。
- Split mode：frontend 負責 UI，backend 負責檔案與 project service。
- 語言格式：JSON、YAML、YML、Laravel PHP 可選 `declare(strict_types=1);` 加上 `return` array，以及 Java ResourceBundle Properties。
- 插件顯示語言：英文、繁體中文、簡體中文、日文、韓文。

## 3. 功能需求

### 3.1 方案管理

- 安裝後在 sidebar/tool window 顯示插件圖示；必須同步提供 16×16、20×20 的 Light／Dark 四種 JetBrains 命名變體，讓 IDE 依主題與縮放自動選用。
- 工具視窗需提供固定 GitHub URL 的「回報問題」外部連結。
- 使用者可新增、切換、刪除與重新讀取方案。
- 新增方案必須輸入名稱，並可選擇一個／多個檔案或一個／多個指定資料夾。
- Tool Window 方案下拉選單必須提供匯入／匯出方案設定；格式為有版本 JSON，包含名稱、列管路徑與使用率設定，不包含 cache、entries 或 issues。
- 匯入必須以目前專案根目錄解析相對路徑，顯示逐檔預覽，並在缺失、不支援或不安全路徑存在時禁止套用。
- 資料夾模式必須先安全遞迴掃描並實際嘗試解析支援格式，再以 popup 列出檔案、格式、locale、namespace、entry 數與錯誤原因；popup 必須允許繼續增加資料夾並合併結果。
- 只有成功識別且由使用者在 popup 勾選確認的檔案可加入方案；掃描結果不得自動納管。
- 多資料夾掃描需限制整批檔案數與各根目錄深度、依正規化完整路徑去重，且略過依賴、建置、版本控制與快取目錄。
- 每個方案只可存取自己的檔案清單。
- 方案可從既有 locale 建立新語言版本；必須保留 key／namespace 結構、清空一般 value、先顯示逐檔 Diff，且不得覆寫既有目標檔案。
- 刪除方案只刪除插件設定與 cache，不刪除來源語言檔。
- 插件不得自動搜尋後納管語言檔。

### 3.2 檔案解析

- JSON：支援巢狀 object、primitive 與 array value。
- JSON array 可包含字串、數字、布林、null、object 與其他 array。
- JSON 句子型 key 的字面句點必須保留，不得誤判為巢狀路徑。
- YAML/YML：支援空白縮排 map、引號及行尾註解；tab 縮排需回報錯誤。
- PHP：僅支援可選的 `declare(strict_types=1);` 與靜態 `return [...]`／`return array(...)`；不得執行 PHP 或函式呼叫。
- Properties：支援註解、`=`／`:`／空白分隔、續行與 Java escape；檔名 locale 後綴與 bundle namespace 必須正確推導。
- 單檔解析失敗不得阻止其他檔案讀取；錯誤需顯示檔名與安全化訊息。

### 3.3 翻譯表

- 以 `namespace + key` JOIN 多語言資料，每個 locale 顯示為獨立欄位。
- 每頁最多顯示 100 個翻譯鍵，提供上一頁／下一頁。
- 支援模糊搜尋、精準搜尋、指定語言篩選，以及 JOIN row 的全部／缺少任一語言翻譯／使用次數為 0 篩選。
- 支援新增、編輯、批量刪除與跨語言 key 改名；新增／編輯必須以可捲動表單同頁列出全部語言，每個 textarea 保留三行／72 px 高度。
- 多語言新增／編輯必須一次驗證全部 mutation 並透過單一批量 RPC 儲存，最後只 reload 一次；部分寫入失敗時必須嘗試還原先前已寫入檔案。
- 選取單一 cell 時，row 操作必須正確映射至該 cell 所屬 row。
- `Ctrl+C` 複製所選 cell；單 cell 不得複製整 row，多 cell 使用 TSV。
- `Ctrl+V` 只允許貼入單一語言 value cell，不得貼入 namespace、key 或診斷欄。
- 可用所選 row 開啟 IDE 原生全文搜尋；所有格式都只搜尋實際 key，不得加入 PHP 檔名或 ResourceBundle bundle 等 namespace。
- 可將目前方案第一個含 `(?<key>…)` 的使用率 Regex 帶入 IDE 全文搜尋：以所選 row 的 literal key 取代 key 群組、移除最外層 `^`／`$`、保留其他群組與反向參照，並啟用 Regex 搜尋模式。
- 多個低頻功能需收進「操作」下拉選單，避免工具列過寬。
- 操作選單需提供「複製 key 到指定語言數值」，支援單列及多列。
- AI Provider 支援 OpenAI-compatible Chat Completions 與 Anthropic Messages API，一次最多翻譯 100 個所選 row；結果需先以可編輯表格及檔案 Diff 預覽，只有 Apply 可寫入。Diff 需提供「提出其他意見 AI」，把來源值、上一輪已檢視翻譯與使用者意見帶入新一輪請求；單筆操作需顯示多選節省 token 的 Toast 提醒。
- AI Token 只可儲存在 JetBrains PasswordSafe，不得寫入一般設定、方案匯出、cache 或 log；非 localhost 端點只允許 HTTPS、禁止 redirect 並限制回應大小。

### 3.4 問題與建議

- 顯示嚴重度、類型、key、訊息、來源檔案與操作欄。
- 偵測項目：
  - `PARSE_ERROR`／`READ_ERROR`
  - `MISSING_VALUE`
  - `DUPLICATE_KEY`
  - `DUPLICATE_VALUE`
  - `MISSING_TRANSLATION`
  - `UNUSED_KEY`
- 操作欄必須為可互動按鈕，其餘診斷欄維持唯讀。
- 支援單列處理、處理所選與處理全部可修復項目。
- 格式／讀取錯誤可開啟來源檔案；需人工判斷的建議可定位到翻譯表。

### 3.5 修復與 Diff 預覽

- 全檔正規化會以 key 補上空白 value，並用標準格式 render。
- 單列空值修復只改動指定 entry。
- 未使用 key 可在問題處理流程中刪除。
- 所有「修復／正規化」與問題自動處理在寫檔前必須顯示雙欄 Diff。
- 批量變更必須能逐檔切換預覽。
- 使用者可選擇套用或取消；取消不得產生任何檔案修改。
- Preview response 必須包含原檔 SHA-256。
- Apply 前必須重建 proposed content 並比對 expected hash；若檔案已變更，必須拒絕覆蓋並要求重新預覽。

### 3.6 使用率

- 每個方案可設定獨立掃描 base path；留白時使用目前 IDE 專案根目錄。
- 使用者可管理 Regex 清單；每個 match 以 `(?<key>…)`、第一個 capture group 或完整 match 擷取候選值，再與 `key`、`namespace.key` 精準比對。
- Regex 編輯器需提供 PHP 主流框架、Spring／Java／Kotlin、ResourceBundle 與 JetBrains Plugin 推薦格式；沒有確認內建翻譯 API 的框架只能提供明確標示的通用自訂 helper 格式。
- 使用次數必須依每個實際 match 累加，包含同一行重複呼叫及不同 Regex 格式的命中；多個 Regex 捕獲同一來源位置的相同 key 時只計一次。
- 新方案排除清單預設包含 `.git`、`.github`、`docs`、`vendor`、`storage`、`database`、`gradle`、`.gradle`、`build`、`out`、`dist`、`target`、`node_modules`、`.idea`、`.run`、`.vscode`、`.fleet`、`.vs`、`.settings`、`.metadata`、`nbproject`、`.env`、`.claude`、`.codex`、`.gemini`、`.agents`、`.ai`，並可自由新增、編輯及刪除資料夾名稱或相對路徑。
- 舊版保留原廠預設的排除清單應自動升級到最新預設；已自訂的清單不得被覆蓋。
- 最多掃描 2,000 個來源檔，單檔不得超過 512 KB。
- 不掃描方案中的語言檔本身。
- Regex、base path 與排除項目須由 backend 驗證；不可接受 URI、device path、控制字元、父目錄 traversal 或無效 Regex。
- 變更掃描設定後必須使該方案 cache 失效並重新計算使用率。
- 解析結果需先推送 UI，使用率掃描不得讓工具視窗長時間無畫面回應。

### 3.7 進階設定

- `Settings → Tools → LanguageManager` 調整插件顯示語言、是否隱藏重複值／可能未使用建議，及新方案預設值；base path 可選目前專案或向上 1–10 層，方案檔案與既有方案掃描設定由 Tool Window 的「方案設定」視窗管理。
- 切換方案時需顯示列管檔案、使用率 base path、Regex 清單及排除清單；各方案草稿互相隔離。
- 舊版方案未包含掃描設定時必須套用安全預設值，不可造成反序列化失敗。

### 3.8 多國語言

- 插件名稱、說明、tool window、按鈕、下拉操作、tab、欄位、分頁、狀態、對話框與 Diff 皆需使用 resource bundle。
- Parser、輸入驗證與分析診斷也需使用 backend resource bundle。
- locale 檔名遵循 JetBrains/Java 標準：base、`_zh_TW`、`_zh_CN`、`_ja`、`_ko`。
- 每個 locale 必須擁有與英文 base 完全相同的 key 集合。

## 4. RPC 與資料需求

### 4.1 狀態模型

`LocalizationStateDto` 必須提供：

- `schemes`
- `activeSchemeId`
- `entries`
- `issues`
- `busy`
- `errorMessage`

Frontend 必須持續 collect backend `state()` flow，不以輪詢更新 UI。

### 4.2 Mutation

- `EntryMutationDto` 必須包含 entry ID（新增時可空）、file path、locale、namespace、key、value。
- Backend 不得相信 frontend 傳入路徑；每次 mutation 皆需重新驗證是否屬於方案。
- 寫入後必須強制 reload，重新建立 entries、issues、usage 與 cache。

### 4.3 Preview

- `ChangePreviewRequestDto` 可表達全檔正規化、指定空值修復與指定 entry 刪除。
- `FileChangePreviewDto` 必須包含 file path、before、after、before SHA-256。
- `applyPreviewedChanges` 必須接收 preview 時的 expected hashes。

## 5. 安全需求

- 檔案路徑不可為空、不得超過 4,096 字元、不得含危險控制字元。
- 僅接受本機一般檔案，不接受 `://`、`ldap:`、`file:`、Windows device path 或 `GLOBALROOT`。
- 僅接受 `.json`、`.yaml`、`.yml`、`.php`、`.properties`，單檔上限 10 MB。
- Value 上限 100,000 字元；key 上限 256 字元。
- Locale 與 namespace 必須符合白名單格式。key 可包含空格、Unicode 與標點，但不可為空白、含控制字元或超過 256 字元。
- PHP parser 不可 eval、include、shell 或呼叫外部程序。
- 輸出需正確跳脫 JSON、YAML、PHP 與 Properties 字串。
- 寫入需使用同目錄 temporary file 與 atomic move；temporary file 最終必須清除。
- 顯示錯誤前需移除不可見控制字元並限制最大長度。

## 6. 快取需求

- 方案清單保存於 `.idea/language-manager/schemes.json`。
- 每個方案使用獨立的 `cache-{schemeId}.json`。
- 記憶體使用 `StateFlow` 保存當前權威狀態。
- Cache 需包含 format version、file fingerprints、entries、issues。
- Cache version 或任一 fingerprint 不符時必須淘汰並重新解析。
- 刪除方案時需刪除對應 cache。

## 7. 非功能需求

- 所有檔案 IO 在 backend IO dispatcher 執行。
- 同一 project service 的 mutation 需以 mutex 序列化。
- UI 更新只在 Swing EDT 執行。
- 大型專案、WSL 或遠端檔案系統不得在 project service constructor 同步走訪檔案。
- RPC DTO 必須使用 `kotlinx.serialization`，不得在 plugin module 內重複打包 serialization runtime。
- Backend RPC provider 不得依賴非必要的 PHP、YAML 或 Properties IDE 插件。

## 8. 驗收測試

- JSON nested object、array、句子型 key 可 round trip。
- YAML nesting、quoted colon、comment 可解析；非法格式可安全回報。
- PHP 靜態 array 可解析；函式呼叫或可執行 expression 必須拒絕。
- Properties 的 base bundle、locale 後綴、escape、續行及 UTF-8 round trip 必須正確；重複 key 與錯誤 escape 必須安全回報。
- URI、device path、unsupported extension 必須拒絕。
- 搜尋、JOIN、locale filter、100 列分頁結果正確。
- 問題分析可找出空值、重複鍵、重複值、缺少語言與未使用 key。
- Preview render 不得修改來源檔。
- 五套 frontend/backend bundle key 必須一致。
- `test` 與 `buildPlugin` 必須成功，ZIP 中需包含三個 module 與所有語言 bundle。

## 9. 不在自動處理範圍

- 自動選取或自動納管專案語言檔。
- 執行 PHP 取得動態翻譯內容。
- 自動刪除無法確定未使用的 key 而不經使用者確認／Diff。
- 對解析失敗的檔案猜測修復後直接寫回。
